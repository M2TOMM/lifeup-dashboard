# LifeUp Dashboard 产品路线图

版本：2026-07-11

## 一句话方向

把 LifeUp Dashboard 完善成一个安全、可靠、适合批量操作的电脑端控制台：本地备份负责完整管理，手机云人升负责实时查看和安全新增，两种数据源始终隔离。

## 当前产品基线

目前已经具备：

- 本地 LifeUp 备份 ZIP 的浏览器导入、工作副本读取、任务/商品/成就等管理与导出。
- 手机云人升连接、实时读取、只读浏览和预览确认后的新增任务。
- 任务、商品、成就、背包、合成、卡池、经济、历史等管理页面。
- 数据源隔离、Token 仅内存保存、云端预览令牌和幂等执行、基础 XSS 防护。
- 20 项自动化回归测试，当前全部通过。

仍然存在四类明显短板：

1. **安全感不足**：当前“快照”只记录文件路径，不是真正的数据副本；批量操作也缺少统一的预览、回滚和操作报告。
2. **批量效率不足**：已有少量批量按钮，但任务、商品、成就、图标还没有形成统一的 CSV/JSON 批量工作台。
3. **洞察页不够真实**：“大道宏愿”“道行复盘”仍使用演示数据，日常页面也有模拟热力图。
4. **交付体验不足**：README、启动说明、工作区清理和桌面版发布流程还没有跟上当前产品。

## 产品原则

- **数据安全优先**：任何新功能都不能以覆盖原始备份或直接改写手机已有数据为代价。
- **先预览，再执行**：批量写入必须显示将修改什么、错误在哪里、执行后结果如何。
- **工作副本优先**：本地管理只作用于工作副本，导出时生成新 ZIP，并验证可恢复性。
- **双数据源隔离**：`local` 和 `cloud` 不自动切换、不静默混合、不做双向同步。
- **真实数据优先**：没有真实字段支持的页面要明确标注演示，不能让用户误以为是 LifeUp 实时数据。
- **小步交付**：每个任务控制在一个专注开发会话内，完成后立即测试和浏览器验证。
- **暂不重写技术栈**：继续使用 Flask + 原生 HTML/JavaScript，先解决产品价值和可靠性，不急于迁移 React 或拆微服务。

## 路线图总览

| 阶段 | 目标 | 用户得到的价值 | 完成标志 |
|---|---|---|---|
| 阶段 0：可信基础 | 确保可恢复、可校验、不会误写 | 敢放心使用本地管理功能 | 真快照、原子导出、批量校验、核心回归齐全 |
| 阶段 1：批量工作台 | 打通任务/商品/成就/图标批量处理 | 大幅减少手机逐条操作 | CSV 模板、预览、执行报告、失败不污染数据 |
| 阶段 2：真实控制台 | 用真实 LifeUp 数据替换演示洞察 | 首页真正回答进度和问题 | 宏愿、复盘、日常热力图均有真实数据来源 |
| 阶段 3：云端体验 | 提升手机连接和新增任务效率 | 更稳定地查看手机并批量新增 | 明确连接状态、渐进加载、分类技能选择、操作记录 |

## 阶段 0：可信基础（最高优先级）

### 0.1 备份 ZIP 安全校验

- 导入前检查 ZIP 结构、数据库文件、路径穿越、体积和解压后大小。
- 无效备份必须在替换当前工作区之前被拒绝。
- 错误提示说明“哪里不符合要求”和“用户下一步怎么做”。

### 0.2 原子导出与恢复验证

- 先写临时文件，验证 ZIP 和 SQLite 完整性后再生成最终导出文件。
- 浏览器流程拒绝把导出目标指向原始备份。
- 导出后自动显示文件路径、大小、校验结果和生成时间。

### 0.3 真正的快照系统

- [x] 将当前只保存路径的“快照”改成真实 ZIP 副本。
- [x] 快照存放在 `workspaces/snapshots/`，包含名称、来源、时间、大小和校验值。
- [x] 恢复快照时先创建新的工作副本，不直接覆盖当前文件。

### 0.4 批量接口严格校验

- 任务和商品批量接口只接受明确允许的操作。
- ID、价格、状态等参数必须在写数据库前完成校验。
- 无效请求返回错误，不能出现“没有修改却提示成功”。

### 0.5 核心流程回归测试

- 覆盖任务、商品、成就的新增、编辑、删除和批量操作。
- 覆盖打开工作副本、导出、重新打开导出文件的完整闭环。
- 每次测试后确认工作副本恢复、原始备份修改时间不变。

阶段验收：用户可以在工作副本中修改数据、创建快照、导出新备份、重新打开验证，并且任何失败都不会破坏此前可用的文件。

## 阶段 1：批量管理工作台（核心产品价值）

### 1.1 统一批量处理框架

所有批量功能共用四步：

1. 选择文件或粘贴数据。
2. 解析并显示预览表格。
3. 标出错误、重复项和将发生的改动。
4. 创建快照后执行，返回逐行结果报告。

### 1.2 任务批量管理

- 提供 CSV 模板下载和 JSON 示例。
- 支持新增、分类调整、技能奖励、金币/经验、频率、冻结/启用等常用字段。
- 支持按标题、分类和字段组合识别潜在重复任务。

### 1.3 商品批量管理

- 支持名称、分类、价格、库存、购买限制和基础效果。
- 编辑已有商品时保留未知的 `extrainfo` 复杂配置。
- 批量改价提供固定值、增加/减少、百分比三种方式和修改前后对比。

### 1.4 成就批量管理

- 支持自定义成就新增、奖励、分类、描述和图标。
- 系统成就继续只读，不能混入写入列表。
- 条件结构无法安全映射时，整行阻止执行并说明原因。

### 1.5 图标资源管理器

- 浏览备份内已有任务、商品、属性和成就图标。
- 支持上传到工作副本、预览、搜索、查找失效引用和批量替换。
- 导出备份时验证图标文件和数据库引用都存在。

阶段验收：用户可以用电脑一次处理几十条任务、商品或成就，并在执行前看懂变化、执行后拿到报告、出现问题时恢复快照。

## 阶段 2：真实目标与复盘控制台

### 2.1 真实“宏愿”映射

- 先使用可配置的任务分类、标签或成就分类映射长期目标。
- 每个宏愿显示总目标、当前进度、近期变化和关联任务。
- 不引入 LifeUp 中不存在的字段；需要额外配置的数据单独保存并明确来源。

### 2.2 真实“道行复盘”

- 使用任务完成记录、番茄记录、金币/经验变化和成就进度生成周报。
- 所有结论都能展开查看对应原始记录。
- 数据不足时显示“暂无足够数据”，不生成随机建议。

### 2.3 日常功课与闭关修炼

- 用真实完成记录替换日常页面的模拟热力图。
- 校准不同 LifeUp 版本的番茄字段映射。
- 支持按日、周、月查看完成率、连续天数和专注时长。

阶段验收：首页、宏愿、复盘和日常页面不再依赖随机数据，所有关键数字都能追溯到本地备份或明确选择的云端数据源。

## 阶段 3：手机云人升体验

### 3.1 连接诊断和渐进加载

- 区分 Host 缺失、端口错误、连接拒绝、超时、Token 错误和数据格式异常。
- 成就等慢接口显示分类级进度，部分分类失败时保留已成功结果。
- 可加入短时内存缓存，但必须显示最后刷新时间和数据来源。

### 3.2 新增任务表单完善

- 从手机实时读取任务分类和技能，改为可搜索选择器。
- 提供 CSV 模板、错误行定位和执行后逐行结果。
- 保留服务端预览令牌、用户确认和幂等键机制。

### 3.3 云端操作记录

- 本地记录每次预览和新增任务的时间、摘要、结果及幂等键摘要。
- 不记录 Token，不把日志当作同步依据。
- 支持查看和导出操作报告，方便排查重复或失败任务。

阶段验收：手机连接失败时用户知道原因；新增任务时不再手填分类/技能 ID；重复点击不会重复创建任务。

## 阶段 4：桌面版与发布

### 4.1 文档和首次使用流程

- 更新 README，准确说明本地备份和手机云端两种模式。
- 增加“第一次打开备份”“安全导出和恢复”“连接手机”的图文步骤。
- 页面内增加数据源、工作副本和未导出修改的明显状态提示。

### 4.2 工作区和日志维护

- 提供可预览的工作副本清理功能，默认保留最近若干份。
- 将日志、工作区、配置、测试产物与源码明确分开。
- 清理前显示文件数量、大小和保留策略，禁止静默删除。

### 4.3 桌面版发布验证

- 更新 pywebview 桌面壳和启动脚本。
- 构建后在干净环境验证打开、导入、修改、快照、导出和重新打开。
- 发布包不包含用户备份、Token、本地配置和开发日志。

阶段验收：新用户只看 README 和页面提示即可完成安全导入、编辑、导出；发布包经过完整冒烟验证。

## 暂时不做

- 不自动同步手机和本地备份。
- 不通过 `/data/import` 写手机。
- 不直接修改手机 SQLite。
- 不编辑或删除手机已有任务、商品和成就。
- 不在没有官方安全接口和独立风险评估前扩展云端写入范围。
- 不为了“技术看起来更现代”重写整个前端框架。

## 产品完成指标

- 原始备份被修改次数：`0`。
- 快照恢复成功率：自动化和人工验证均为 `100%`。
- 批量操作：执行前有预览，错误行不会写入，执行后有逐行报告。
- 数据来源：所有页面都明确显示 `local`、`cloud` 或“演示数据”。
- 核心管理流程：任务、商品、成就、图标均有至少一个端到端回归测试。
- 发布前：完整单元测试、Python/JavaScript 语法检查、`git diff --check` 和浏览器冒烟测试全部通过。
