# ADR-004：桌面用户数据目录与托管清理边界

## 状态

Accepted

## 日期

2026-07-18

## 背景

PyInstaller 单文件桌面包会把程序资源解压到临时目录。若工作副本、快照、导出和配置继续相对 `server.py` 保存，这些数据会落入临时解压目录，退出后不稳定。与此同时，项目需要清理长期积累的旧工作副本和日志，但不能误删原始备份、当前工作副本、快照或用户普通文件。

## 决策

1. 浏览器开发版默认继续使用项目目录作为数据根目录。
2. 桌面版在导入 `server.py` 前设置 `LIFEUP_DASHBOARD_DATA_DIR=%LOCALAPPDATA%\LifeUpDashboard`。
3. 工作副本、恢复副本、快照、导出、目标映射、云端非敏感配置和操作日志均从该数据根目录派生；Token 仍只在进程内存中。
4. 清理预览只扫描四类托管候选：符合项目命名规则的旧浏览器导入副本、旧恢复副本、自动导出，以及 `work/` 中的临时文件。
5. 快照目录不参与维护页清理。当前来源、原始备份、受保护路径、符号链接和不符合命名规则的文件永不列入候选。
6. 删除必须使用 10 分钟一次性预览令牌和不透明项目 ID；执行前重新校验路径、文件身份、大小和修改时间。

## 备选方案

### 清理整个 `workspaces/` 或按天数自动删除

实现简单，但可能删除当前工作副本或用户需要的恢复点，也缺少明确确认，因此拒绝。

### 允许用户输入任意路径清理

灵活但扩大路径穿越、链接跳转和误删风险，因此拒绝。

### 把桌面数据保存在 EXE 同目录

绿色软件看起来直观，但 Program Files、下载目录或只读位置可能无写权限，也容易把个人数据误打入发布包，因此拒绝。

## 影响

- 桌面用户数据在应用升级后仍保持稳定。
- 发布包只需要包含 EXE 和文档，不需要携带任何用户数据目录。
- 清理流程多一步预览和确认，但删除边界可解释、可测试，并能抵抗预览后的文件替换。
- 旧版位于用户主目录的 `.lifeup_dashboard_config.json` 不会自动迁移；它只保存历史路径，可由用户手动删除。
